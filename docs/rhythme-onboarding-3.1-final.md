# Rhythmé Onboarding 3.1 — Final Decision Document

**Date:** 26 May 2026
**Status:** Locked. Execute exactly. No further iteration until after first users.

---

## 1. Final Vision

**One Primary Goal Only.**
This becomes the single workspace anchor. Everything in the app — tasks, habits, logging, Now Panel, future insights — orbits this one goal.

- No multi-goal complexity in MVP.
- User enters exactly one high-level goal (title + optional description).
- Delivers maximum clarity and emotional focus. Aligns with PRD v3 principles.

**Workspace Concept:**
The entire user session and dashboard = "Your [Goal Title] Workspace." All tasks and habits are linked to this single `user_goal`. Future phases (V2+) can introduce supporting goals without breaking the primary anchor.

**Sub-Goals:**
Do NOT generate sub-goals with AI during onboarding. Tasks generated by 6.9 act as sub-goals (especially for vague goals). This is simple and sufficient for MVP. Defer hierarchical sub-goal / decomposition logic to post-MVP (NBAE era).

---

## 2. Final Flow (Step Order Is Non-Negotiable)

```
Step 1 — Account Created
         ↓
Step 2 — Single Goal Input          ← FIRST. Before profile.
         "What are you building toward?"
         Title (required) + Description (optional)
         CTA: "Let's Build This" [see Section 5 for alternatives]
         ↓
Step 3 — Quick Profile
         Name + daily task target + daily habit target
         [Now has context — user has already stated their goal]
         ↓
Step 4 — Generation Loading
         Intentional state — not a blank spinner.
         Copy: "Analyzing your goal and building your first execution plan."
         [834ms average — Groq 70B. Do not show a raw spinner.]
         ↓
Step 5 — Edit Screen
         Generated tasks (3–5) + habits (2–4)
         Full inline editing.
         Regenerate button — max 2 uses.
         After 2nd regeneration: soft message shown —
         "Your plan looks solid. You can adjust tasks anytime after setup."
         [Prevents anxiety loop. Protects API credits.]
         ↓
Step 6 — Commitment Screen          ← NEW. Non-negotiable.
         Goal title displayed large.
         Line 1: "Your workspace is ready."
         Line 2 (smaller): "The longer you log, the better Rhythmé
                            understands how you work."
         Single CTA: "Start Building →"
         Auto-advance after 3 seconds if no interaction.
         [This is the psychological handoff. User moves from
          'setting up a tool' to 'working toward something'.]
         ↓
Step 7 — Dashboard
         Now Panel populated from this goal.
         Tasks and habits visible immediately.
```

**Why Goal before Profile:**
Asking for daily targets before the user has stated their goal makes the first interaction feel like a settings form. After they type their goal, every subsequent question has meaning. "How many tasks per day?" lands differently once someone has typed "Launch my first product in 3 months."

## 3. Final Schema

### `user_goals` table (new — required)

```sql
create table public.user_goals (
  id            uuid        default gen_random_uuid() primary key,
  user_id       uuid        not null references auth.users(id) on delete cascade,
  title         text        not null,
  description   text,
  is_primary    boolean     default true,
  created_at    timestamptz default now(),
  updated_at    timestamptz default now(),
  metadata      jsonb       default '{}'::jsonb,
  constraint user_goals_user_id_key unique(user_id)  -- one goal per user in MVP
);

create index idx_user_goals_user on public.user_goals(user_id);
```

### Updates to `tasks` table

```sql
alter table public.tasks
  add column goal_id uuid references public.user_goals(id),
  add column source  text default 'manual'
    check (source in ('ai_generated', 'user_edited', 'manual'));
```

**Why `source` matters:**
If the user edits an AI-generated task, that needs to be tracked. Future 6.4 analysis can answer: "do AI-generated tasks get completed more often than manually edited ones?" This is a one-column addition now that would require a migration and backfill later. Add it now.

### Updates to `habits` table

```sql
alter table public.habits
  add column goal_id uuid references public.user_goals(id);
```

### Updates to `user_preferences`

```sql
alter table public.user_preferences
  add column onboarding_completed boolean default false;
```

### RLS Policies for `user_goals`

```sql
alter table public.user_goals enable row level security;

create policy "Users can view their own goal"
  on public.user_goals for select
  using (auth.uid() = user_id);

create policy "Users can insert their own goal"
  on public.user_goals for insert
  with check (auth.uid() = user_id);

create policy "Users can update their own goal"
  on public.user_goals for update
  using (auth.uid() = user_id);
```

## 4. Migration Script for Existing `onboarding_data`

Run once. Handle all three cases explicitly — do not decide under pressure at execution time.

```sql
-- Case 1: onboarding_data has a valid goal title → migrate cleanly
insert into public.user_goals (user_id, title, description)
select
  user_id,
  coalesce(
    onboarding_data->>'goal_title',
    onboarding_data->>'primary_goal',
    'My Goal'           -- Case 2: data exists but no extractable title → placeholder
  ) as title,
  onboarding_data->>'goal_description' as description
from public.user_preferences
where onboarding_data is not null
  and onboarding_data != '{}'::jsonb
on conflict (user_id) do nothing;

-- Case 3: onboarding_data is null or empty → skip silently
-- These users will be re-prompted at next login via onboarding_completed = false
```

**Fallback behavior defined:**
- Valid title found → migrated as-is.
- Data exists but malformed → placeholder title "My Goal" inserted. User sees a pre-filled goal they can edit.
- No data → skipped. User hits onboarding flow again on next login.

## 5. Button Copy — "Let's Build This"

**Primary recommendation:** `"Let's Build This"`

This is the warmest option. "Let's" implies the product is doing this *with* the user, not *for* them. "Build" speaks directly to the beachhead (solo founders, indie builders). It has energy without being cold or clinical.

**Alternatives if tone needs adjustment:**
- `"Build My Plan"` — clean, direct, slightly less warm
- `"Make It Real"` — emotional, high energy, bolder
- `"Start Building →"` — minimal, action-forward

**Rejected:** `"Structure My Goal"` — clinical, enterprise SaaS feeling, does not match premium calm tone.

All other CTAs in the flow:
- Profile step next button: `"Continue →"`
- Edit screen confirm: `"This Looks Right →"`
- Commitment screen CTA: `"Start Building →"`

## 6. What Gets Stored at Completion

When the user hits "Start Building →" on the Commitment Screen, execute in order:

1. Insert into `user_goals` — title, description, `is_primary: true`.
2. Insert tasks into `tasks` — each with `goal_id`, `source: 'ai_generated'`. If user edited any task before saving, set that task's `source: 'user_edited'`.
3. Insert habits into `habits` — each with `goal_id`.
4. Set `onboarding_completed = true` in `user_preferences`.
5. Redirect to Dashboard.

**On failure:** If the Supabase inserts fail, do not drop the user on a blank screen. Show a soft error: "Something went wrong saving your plan. Try again?" with a retry button. Never lose the generated output — keep it in component state until confirmed saved.

## 7. What the 6.9 Endpoint Returns

The existing endpoint `{ML_ENDPOINT}/v1/onboarding/generate` is already built and tested. Do not modify it. Consume it exactly as specced.

**Request:**
```json
{
  "goal_title": "string",
  "goal_description": "string"
}
```

**Response:**
```json
{
  "tasks": [
    {
      "title": "string",
      "description": "string",
      "type": "starter"
    }
  ],
  "habits": [
    {
      "title": "string",
      "frequency": "daily | 2x per week | 3x per week | once per week",
      "reason": "string"
    }
  ],
  "generated": true,
  "fallback_used": false
}
```

If `fallback_used: true`, the UI shows the same edit screen. No error state. The fallback output is valid and usable.

## 8. What Is Out of Scope for This Build

Hard exclusions. Do not build, design, or spec these until after first users:

- Multiple goals or goal switching UI
- Sub-goal hierarchy or decomposition
- Goal progress percentage or completion tracking
- AI-powered goal editing or re-prompting mid-flow
- Social or sharing features
- Any analytics or insight cards during onboarding
- Mobile-native gestures or swipe navigation

---

## 9. Implementation Sequence

**Do these in order. Do not parallelize schema and frontend.**

1. Run schema changes (new table + columns + RLS).
2. Run migration script for existing `onboarding_data`.
3. Generate TypeScript types from updated Supabase schema.
4. Build onboarding components against the new types.
5. Connect generation + save logic to new schema.
6. Update Dashboard / Now Panel to load from primary goal + linked tasks/habits.
7. Test full flow end-to-end: new user → goal → generate → edit → commit → dashboard.

| `source` field on tasks | Required. Add now or regret at 6.4 analysis time. |

| Loading copy | "Analyzing your goal and building your first execution plan." |
| Regenerate limit | 2 uses. Soft message after second use. |
| Commitment screen | Required. One behavioral intelligence line included. |
